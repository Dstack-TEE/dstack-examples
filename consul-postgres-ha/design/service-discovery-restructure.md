# Design: service-discovery restructuring â€” `service:port` UX via per-service VIPs

**Status**: design accepted, not started. Single rewrite, no
staging â€” old impl is committed, rollback is `git revert`, nothing in
production. Branch off `dstack-consul-ha-db`, PR back into it.

**Dependencies**: builds on the TF-broadcast secrets pattern from
commit `e2286a9` (gossip key + Patroni passwords as cluster-wide
identical material via `random_bytes` env). Attestation-rooted
admission is **explicitly the last** architectural stage, not a
prerequisite â€” this design must work end-to-end without it.
Intentions remain plain service-name based until then.

## Why

App-side connection targets today encode peer ordinals in the port
number: to reach worker-3's Postgres an app dials `127.0.0.1:18703`,
to reach worker-4's it dials `127.0.0.1:18704`, computed as
`base + ordinal`. That means every consumer hard-codes "which peer
holds which role" â€” there is no Compose/K8s-style `postgres:5432`
address that means "the right Postgres, wherever it is right now".

It also leaks through the platform plumbing in two places:

- `PEERS_JSON` in `compose/worker.yaml` carries an 8-port array per
  peer (one per protocol slot), shared across the cluster, must
  round-trip identically.
- mesh-conn's per-peer-port arithmetic and `self.hasPort` lookup
  dispatch on those slots.

Patroni leader changes are invisible at this layer â€” Patroni handles
master election but a client connecting to `127.0.0.1:18703` lands on
worker-3's Postgres regardless of whether worker-3 is the current
leader. Today's worked example doesn't notice because every CVM only
runs one consumer (webdemo) and the demo tolerates "wrong instance".
A real workload would not.

The fix is to invert the addressing axis: **ports identify services,
peers are invisible to the app.**

## Goal

After this work:

- Apps reach services by name + well-known port: `postgres-rw:5432`,
  `webdemo:8080`. Same string on every CVM. No ordinals.
- Patroni failover transparently retargets `postgres-rw` consumers
  via Consul EDS push â€” no app retry against another ordinal.
- mesh-conn drops all knowledge of services and protocols. Its
  config shrinks to "for each peer, here's its identifier and its
  loopback VIP". The 8-protocol-slot array goes away.
- Connect intentions become the actual policy enforcement boundary
  (deny-by-default + per-pair allow), independent of attestation
  work which lands later.

## Non-goals

- **Replacing mesh-conn's transport layer.** ICE+QUIC+pair-wise links
  stay exactly as they are. This work changes what mesh-conn
  *exposes* on each CVM, not how it carries bytes between them.
- **Attestation-rooted intentions.** Out of scope here; lands as a
  later step on top of stock Connect intentions. See
  [`attestation-admission.md`](attestation-admission.md).
- **Cross-CVM Postgres failover RTO improvements.** Patroni's
  `ttl=30` still bounds the lower edge. EDS push only ensures the
  *consumer* sees the new leader within ~1s of Consul knowing.
- **Replacing mesh-conn with a TUN-based virtual L3 overlay.** The
  loopback-only invariant is intentional (rejected once already).

## Two parallel addressing namespaces

Two `/24`s carved out of the loopback `127/8` block. They never
overlap and grow independently.

| Range | Purpose | Allocation |
|---|---|---|
| `127.10.0.0/24` | **Service VIPs** â€” one per Connect upstream a CVM consumes | per-CVM, derived from that CVM's app's upstream list |
| `127.50.0.0/24` | **Peer VIPs** â€” one per remote peer | per-CVM, derived from PEERS_JSON; ordinal â†’ last octet |

Loopback supports thousands of aliases; cost is negligible. Both
ranges are provisioned by `mesh-sidecar/entrypoint.sh` at startup
via `ip addr add 127.x.0.<n>/32 dev lo`.

`/etc/hosts` on each CVM maps each service name to its service VIP:

```
127.10.0.1  postgres-rw
127.10.0.2  postgres-ro
127.10.0.3  webdemo
```

Generated by the entrypoint from the upstream list. Apps then call
`postgres-rw:5432` and `getaddrinfo` returns `127.10.0.1`.

## Static port allowlist on peer VIPs â€” platform infrastructure only

The set of ports any host ever dials at a peer VIP is fixed, small,
and entirely platform infrastructure:

| Port | Used by | Proto |
|---|---|---|
| 21000 | Remote Envoy public mTLS listener | TCP |
| 8300 | Consul server RPC (coordinator peers only) | TCP |
| 8301 | Consul serf-LAN gossip | UDP+TCP |

mesh-conn binds `(127.50.0.<peer>, port)` for each `(peer, port)` in
the allowlist at startup. ~9 listeners per worker, all known
statically.

**No application-level traffic in the allowlist.** This example is a
vehicle for a general service mesh, not a Postgres-specific
deployment. Patroni's streaming replication goes through Envoy as a
stock Connect upstream like any other consumer (`postgres-master`),
not via a peer-VIP bypass. The 2Ã— Envoy hop on `pg_basebackup`
(~25 MB/s sustained) is the cost of uniformity and is accepted: a
single transport story for all app traffic, no per-workload
allowlist entries.

Apps never dial peer VIPs. Only Envoy and Consul-agent do, and both
speak well-known platform ports. This eliminates the dynamic-port-
binding problem that the earlier TPROXY sketch tried to solve: the
static set works, so iptables complexity isn't earning anything.

If a future workload genuinely needs a high-throughput internal
channel that bypasses Envoy, it's a code change in
`mesh-conn/main.go` plus a justification in this doc â€” mesh-conn
does not pre-anticipate that.

### Layering invariant (kept)

mesh-conn knows peers, not services. The allowlist is small,
declarative, and lives in the transport layer. There is no
catalog-watch in mesh-conn; nothing in mesh-conn looks at Consul.

### Well-known ports as a structural invariant

The static allowlist depends on every CVM binding the same port for
the same role: Postgres on `:5432`, Consul serf on `:8301`, Envoy
public on `:21000`. This is safe by construction because:

- `network_mode: host` per CVM, so the bind is per-CVM, not per-host.
- Loopback-only â€” services bind `127.0.0.1:<port>`, never `0.0.0.0`,
  so there's no risk of one app's listener shadowing another's.
- One instance of each role per CVM. If we ever needed two Patronis
  on one CVM, the design breaks; we don't, and won't.

`compose/worker.yaml` and `patroni/entrypoint.sh` both shed their
`base+ordinal` arithmetic in Stage 2 â€” Postgres listens on `:5432`
on every worker, period.

## Envoy on the data path

### Bootstrap

Use Consul's **built-in CA provider**. Servers generate the CA root
in Raft on first quorum â€” no external KMS dependency, no per-CVM
derivation problem. Connect CA root is *not* one of the secrets
broadcast via TF env (unlike gossip-key + Patroni passwords per
commit `e2286a9`); the cluster mints it itself.

Worker entrypoint ordering:

```sh
mesh-conn &                              # transport layer up
consul agent &                           # joins gossip, sees quorum
until consul members | grep alive; do sleep 1; done
consul connect ca leaf -service=$SELF >/dev/null   # blocks until issuable
consul connect envoy -sidecar-for=$SELF
```

The `ca leaf` call is the synchronization gate. It returns once a
server can mint a leaf cert, which is precisely the condition "Envoy
can start now". Servers themselves form quorum without Envoy
because Consul-server RPC rides peer VIPs as plain TCP through
mesh-conn, no mesh required.

### Degraded-Envoy recovery

Run Envoy under a tiny supervise loop in `entrypoint.sh`:

```sh
until consul connect envoy -sidecar-for=$SELF; do sleep 1; done
```

Liveness from Envoy admin `/ready`, *not* from a Consul check that
would deregister the sidecar service (which would feedback-loop into
mesh-wide endpoint churn). On a crash, apps see RST during the ~2s
restart window â€” their existing retry handles it.

`proxy-defaults` config entry sets per-cluster circuit-breakers
(`max_pending_requests`, `retry_budget`) so a downstream that's slow
doesn't take out the local Envoy.

**Crucially**: Consul HTTP API stays at `127.0.0.1:8500` on every
CVM, **not** in the mesh. Apps that read KV/catalog use the local
agent directly. A degraded local Envoy doesn't blind the local app
from the Consul control plane.

## Implementation â€” single rewrite

The old per-port-per-peer scheme is committed but not in production;
rollback is `git revert`. No staging or coexistence machinery.
Commits on the new branch can be small for review hygiene, but the
design does not maintain backward compatibility with the old
`base+ordinal` PEERS_JSON shape.

**Code changes**, by file:

- `mesh-sidecar/entrypoint.sh`:
  - Provision `127.50.0.<n>/32 dev lo` for each peer in PEERS_JSON.
  - Provision `127.10.0.<svc>/32 dev lo` per declared upstream.
  - Render `/etc/hosts` from the upstream list (service-name â†’ service-VIP).
  - Replace existing Envoy launch with the supervise loop +
    `consul connect ca leaf` synchronization gate documented above.

- `mesh-conn/main.go`:
  - Drop per-protocol-port arithmetic and `self.hasPort` lookup.
  - Bind `(127.50.0.<peer>, port)` for each port in the static
    allowlist `{21000, 8300, 8301}` per remote peer.
  - 3-byte stream header keeps `(tag, port)` shape; receiver
    dispatches by dialing `127.0.0.1:<port>` directly, with the
    allowlist as the validation set.
  - PEERS_JSON shape: `[{id, vip-octet}]`. Per-peer Ports arrays
    gone.

- `compose/worker.yaml`:
  - Drop per-protocol port arithmetic from the env block.
  - PEERS_JSON shrinks accordingly.

- `cluster-example/cluster.tf`:
  - Declare service-VIP allocations + Connect upstream blocks per
    app:

    ```hcl
    upstreams = [
      { destination_name = "postgres-master", local_bind_address = "127.10.0.1", local_bind_port = 5432 },
      { destination_name = "postgres-replica", local_bind_address = "127.10.0.2", local_bind_port = 5432 },
    ]
    ```

  - Apply config entries via `consul config write`:
    - `proxy-defaults` (tcp default).
    - `service-resolver postgres-master` with
      `Failoverâ†’postgres-replica`.
    - `service-intentions` `*` deny first, per-pair allows after.

- `patroni/entrypoint.sh`:
  - Postgres listens on `127.0.0.1:5432` on every worker (no
    `base+ordinal`).
  - DCS URL hardcoded `127.0.0.1:8500` (drop `info.json`-driven
    `CONSUL_PORT` lookup).
  - `replicatefrom` / `pg_basebackup` source = `postgres-master:5432`
    (resolves via `/etc/hosts` to local service VIP, then through
    local Envoy via Connect upstream).

- `validate_test.go`:
  - Updated to reflect the shrunk PEERS_JSON shape.

## Open questions

1. **Allowlist evolution path.** When (not if) a future service
   needs the allowlist extended, how do operators discover that
   without reading the source? Consider emitting the active
   allowlist on mesh-conn startup with a comment pointing at this
   doc.
2. **Loopback alias provisioning capability.** `ip addr add` needs
   `NET_ADMIN` in the sidecar container. Today the sidecar already
   has it for mesh-conn's UDP socket setup â€” verify, document.
3. **`proxy-defaults` vs per-service `service-defaults`.** The
   default `protocol = "tcp"` is right for Postgres but wrong if
   we ever want intention-on-path enforcement for HTTP services.
   Per-service overrides handle this; mention in success criteria.
4. **EDS update latency under Patroni failover.** Patroni's
   register/deregister of `<scope>-master` happens on its check
   interval (we use `service_check_interval=5s`). Verify the
   end-to-end window from failover-trigger to consumer-Envoy-using-
   new-endpoint is bounded by `5s + xDS push latency`, not anything
   worse.

## Success criteria

- [ ] On any CVM, an app can `psql -h postgres-master -p 5432`
      successfully and the connection lands on the current leader.
- [ ] After `patronictl failover`, the same `psql` command run within
      ~1s lands on the new leader, no client-side retry needed.
- [ ] After `consul config write` of a deny intention from `webdemo`
      to `postgres-master`, the connection from webdemo is refused.
- [ ] `mesh-conn/main.go` line count drops by â‰¥25%.
- [ ] `PEERS_JSON` schema in `compose/worker.yaml` no longer carries
      protocol-port arrays â€” peers are `{id, vip-octet}`, no per-peer
      port lists.
- [ ] mesh-conn allowlist contains exactly `{21000, 8300, 8301}` â€”
      no application-level ports.
- [ ] All `FAILOVER.md` recipes still pass with measured RTO within
      noise of the pre-restructure baseline.
- [ ] Patroni `pg_basebackup` rebuild (disk-loss recovery from
      `FAILOVER.md`) completes through Envoy at acceptable
      throughput. Document the measured rate; acknowledge the
      Envoy-hop tax explicitly.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| EDS endpoint update doesn't propagate fast enough on failover, app sees stale leader for >5s | Measure in Stage 2. If too slow, drop `service_check_interval` to 2s and verify Patroni doesn't false-flap under load. Consul push latency is in milliseconds; the slow link is Patroni's check interval. |
| Loopback aliases don't survive sidecar container restart, mesh breaks until re-provisioned | Aliases live in the network namespace of the sidecar container, which is shared via `network_mode: host` â€” so restarts don't lose them; only host reboot does, and then `entrypoint.sh` re-provisions. Verify in Stage 1. |
| `consul config write` of intentions racing with first apply leaves a window where everything is allowed | Apply `*` deny intention as the *first* config-entry write in `cluster-example/cluster.tf`. Per-pair allows come after. |
| Built-in CA root reset on `terraform destroy + apply` invalidates all in-flight Envoy certs | Document in `FAILOVER.md` â€” full cluster destroy/recreate triggers a CA rotation; existing Envoys must restart to pick up new leaf certs. Acceptable; matches how Connect normally bootstraps. |
| Patroni `consul.url` doesn't accept a non-default port cleanly | Verified by reading current `patroni/entrypoint.sh` â€” already passes `host:port` through. Stage 3 just hardcodes the same shape with a different value. Low risk. |
| Patroni `pg_basebackup` throughput drops below usable through 2Ã— Envoy hop | Measure early. If unacceptable, reconsider whether replication should add an exception to the allowlist â€” but only with explicit doc justification, not as a default. |
